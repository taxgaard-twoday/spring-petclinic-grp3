# Repository Instructions

## Branching

For each new implementation iteration, create a feature branch before making code changes.

Use a short descriptive branch name, for example:
`feature/appointment-availability-api`

Before creating the branch:
- check the current branch and working tree status
- do not discard or overwrite existing user changes
- if there are uncommitted changes from a previous iteration, ask before branching unless the user has clearly said to continue from the current state

Do not create a branch for read-only exploration, planning, review, or answering questions.
